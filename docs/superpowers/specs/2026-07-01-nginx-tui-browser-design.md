# nginx 目录浏览下载 TUI — 设计文档

日期：2026-07-01

## 概述

一个单文件 Python 脚本，接收一个开启了 `autoindex on;` 的 nginx 静态文件服务器 URL，在终端里以全屏 TUI 浏览其目录结构，支持进入子目录、返回上一级、以及回车/鼠标点击下载文件到本地指定目录，下载时显示进度。

## 目标 / 非目标

**目标**
- 浏览 nginx autoindex 生成的目录列表，支持进入子目录、返回上一级
- 回车或鼠标左键点击文件条目触发下载，显示下载进度
- 展示的文件名/大小/修改时间忠实于 nginx 返回的真实数据；列表的排版/配色是脚本自己设计的（不照抄原始 `<pre>` 文本排版）
- 零第三方依赖，单文件即可运行（Python 3 标准库：`curses`/`urllib`/`html.parser`/`argparse`）

**非目标**
- 递归下载整个目录
- 需要登录认证（Basic Auth / Cookie）的服务器
- 自定义 CA 证书的 HTTPS（默认使用系统默认 SSL 上下文；`--insecure`/`-k` 只是跳过证书校验，不做自定义 CA 信任链）
- 排序/过滤/搜索等高级列表操作
- 断点续传

## 设计原则

以下几条贯穿全篇，是具体行为背后共同的"为什么"。后面各节描述具体行为时不再重复讲道理，只在需要时反过来引用这里；新增功能时也应该先对照这几条判断该怎么做，而不是每次临时决定。

**1. UI 响应速度最高优先**——优先级高于代码简洁度这类其他非功能性考量。任何让用户觉得"卡住了/没反应"的行为都按 bug 处理，哪怕背后有正当理由（网络本来就慢、这是 ncurses 默认行为、只是极小概率的边界情况）。
  - `ESCDELAY` 通过 `setdefault` 由 ncurses 默认 1000ms 降到 25ms（只设默认值，不覆盖用户已配置的值），否则单独按 `Esc`（返回上级）会有明显的可感知延迟
  - `curses.mouseinterval(0)`，关掉 curses 默认 200ms 的"按下+松开合成一次点击"等待——`_handle_mouse` 只看按下事件就够，不需要等这个窗口
  - 任何阻塞在网络 I/O 上的操作（加载/刷新目录、下载文件）在发起请求前先同步显示状态提示，不等到有结果或第一次进度回调才反馈
  - `stdscr.timeout(100)` 让主循环定期重绘，临时状态提示能自己按超时消失，不需要等下一次按键
  - 新增任何会让用户等待、且等待期间界面没反馈的操作，先问"用户能不能看出程序在做事、能不能取消"，答案是否定就要照上面的模式处理，不要等用户反馈"感觉卡"了再补

**2. 不可信文本渲染前必须净化控制字符**——服务器返回的文件名、路径，以及网络异常的错误消息，都可能带着裸控制字节（换行、制表符等）；这些字节一旦被 curses 当成真实的光标移动指令，会把排版拆乱。
  - 统一替换成占位符 `�`（`_sanitize_display_text`）：覆盖条目名、面包屑、状态栏文字——包括异常的 `str(exc)`（HTTP 状态行的 reason phrase 本身允许带字面制表符）、mtime 解析失败时保留的原始字符串兜底
  - 同一套净化结果也直接用作本地落盘文件名（不单独区分"显示用"和"落盘用"两条路径），避免服务器文件名里的裸控制字节被原样写进本地文件系统

**3. 任何进入 curses 的文本都要按可用宽度裁剪**——不管内容是否可信，超宽都会撑爆列宽或把关键信息挤出屏幕；这一条和内容来源无关，程序自己的固定文案（比如按键提示行）在窄终端下一样会超宽。
  - 裁剪成"保留头尾、中间省略号"，而不是简单掐尾：面包屑、状态栏临时提示、常驻按键提示行的结尾经常带着关键信息（最深一级目录名、"(y/n)"确认、Ctrl-C 取消提示、"q/Q 退出"），纯掐尾会把这些挤没
  - 定长列（大小、修改时间）只裁剪不换行，一行内容超宽会把整行内容挤到下一行、弄乱后续所有行的排版

**4. 涉及覆盖/删除/提交数据的操作不静默执行**——宁可多弹一次确认或直接报错拒绝，也不做用户没确认过的破坏性操作。
  - 目标路径或其 `.part` 暂存文件已存在：弹出覆盖确认，用 `lexists` 而非 `exists` 判断（`exists` 对失效的软链接返回 `False`，会让悬空软链接绕过确认直接被覆盖）
  - 目标路径已经是目录：直接拒绝，不进覆盖确认流程——否则要等一整个文件下载完、最后 rename 才失败，白白浪费一次完整下载
  - 下载数据不完整（已接收字节数与 `Content-Length` 不符）：报错，不当成功提交——连接提前断开时 `HTTPResponse.read()` 只会悄悄返回空字节而不抛异常，不做这个校验会让残缺文件被当成下载成功
  - 下载失败/中断时清理 `.part` 残留，但只在本次下载确实创建/打开过它的前提下才清理，不误删本机已存在、并非本次下载产生的同名文件

**5. 局部失败要局部降级，不拖垮整体**——单个条目、单帧绘制、单次非核心初始化调用失败，不应该让程序崩溃或丢失用户已有的浏览状态。
  - 单个条目日期解析失败：该条目退化显示原始字符串，不影响其他条目和整体加载
  - 单帧绘制越界抛出的 `curses.error`：跳过这一帧，下一次重绘再试，不影响程序继续运行
  - `locale.setlocale`、`curs_set` 等次要初始化调用失败：忽略，不阻断启动
  - 加载/刷新期间 Ctrl-C 取消：保留取消前的旧状态（历史栈、当前列表），不清空

## 架构与组件

- `Entry`（dataclass）：`name`（`unquote(href)` 后按原则 2 净化控制字符；目录含尾部 `/`）、`href`（原始链接，用于拼 URL，不做净化——净化只影响显示和本地落盘文件名，不影响请求路径）、`url`（与当前页面 base URL 拼接后的绝对地址）、`is_dir`（bool）、`size_bytes`（`Optional[int]`，精确字节数；目录或非精确大小时为 `None`）、`mtime`（`Optional[str]`，格式化为 `YYYY-MM-DD HH:MM`；解析失败保留原始字符串兜底，同样按原则 2 净化）、`size_raw`（`Optional[str]`；仅服务端给出 `autoindex_exact_size off` 那种四舍五入形式如 `24M`/`1.2K` 时才有值，原样展示不换算字节数）
- `parse_index(html, base_url) -> List[Entry]`：
  - 用 `html.parser.HTMLParser` 提取所有 `<a href="...">`，不依赖服务端返回文本的具体格式；未闭合就被新 `<a>` 打断的畸形嵌套标签，会先把还没收尾的外层锚点原样记下，再开始追踪新的一层，不静默丢弃
  - 一个 `<a>` 都提取不到时返回空列表（等同空目录），不报错——不同服务端对"空目录"的表示不一致（nginx 空目录仍带 `../` 链接，Python `http.server` 生成的是零 `<a>` 标签），无法用链接数量可靠区分"真的是空目录"和"根本不是目录列表页"
  - 名称统一取 `unquote(href)`，不用链接文本——超长文件名 nginx 会在链接文本里截断显示（`..&gt;`），href 才是完整的
  - 跳过父目录项（`href` 为 `../`、`..`、`/` 或链接文本为 `..`），返回上级由脚本自己的历史栈管理
  - 每个 `<a>` 所在的原始文本行，用正则尽力解析其后跟着的日期（`%d-%b-%Y %H:%M`）和大小；大小允许纯数字（精确字节数，写入 `size_bytes`）或四舍五入单位形式（写入 `size_raw`，原样展示）；日期或大小整体解析不到就置空，不影响该条目显示和下载；这一步按正则原文抓的 `href` 会先做一次 HTML 实体反转义，再去匹配锚点提取阶段（`HTMLParser` 已自动解码）拿到的 `href`，否则页面把 `&` 写成 `&amp;amp;` 这类二次转义会查不到对应的 size/mtime
- `fetch_index(url, insecure=False) -> (html, final_url)`：`urllib.request` 拉取 HTML，设超时，返回重定向后的最终 URL（nginx 对缺尾部斜杠的目录请求会 301 重定向）——调用方必须用它作 `parse_index` 的 base URL 和历史栈存的地址，不能用请求时的原始 URL；响应体超过 10MB 直接报错放弃，防止一个实际不是目录列表的超大响应把内存吃满卡死界面；HTTP/网络错误交给调用方处理；`insecure=True` 时用不校验证书/主机名的 SSL 上下文
- `download_file(url, dest_path, progress_cb, insecure=False)`：流式分块（64KB）写入 `dest_path + ".part"`，成功后 rename 成目标文件名；数据完整性、`.part` 清理规则见原则 4；`Content-Length` 为负数（畸形响应）时当作没有这个头处理，否则完整下载也会被完整性校验误判成"不完整"

## 导航：历史栈

`BrowserApp` 维护一个栈，每个栈帧是 `(url, entries, selected_index, scroll_offset)`：
- 进入目录：`fetch_index` + `parse_index` 后压入新栈帧
- 返回上级（`Backspace`/`←`/`u`/`Esc`）：弹出当前栈帧，直接用上一帧缓存的 `entries` 重新渲染，**不重新发请求**，并恢复之前的光标位置
- 栈只剩一帧（初始 URL）时再按返回：状态栏提示"已在根目录"，不做操作

## 界面与展示

- 顶部：当前路径面包屑（对当前 URL 做 percent-decode 以可读原样显示中文等字符）；超宽从中间省略（原则 3，保留开头的协议/主机和结尾最深一级目录名）；有颜色支持时用固定的白字蓝底配色（而不是对终端默认色做反色，避免暗色主题下对比度不足），无颜色支持时退化为 `A_REVERSE`
- 列表区：固定列布局，自己设计的配色和对齐方式——
  - 名称列：自适应宽度，超长按脚本自己的规则截断
  - 大小列：右对齐，`size_bytes` 换算成人类可读单位；没有精确字节数但有四舍五入形式时显示 `size_raw` 原文；都没有或是目录显示 `-`；超长裁剪到列宽（原则 3）
  - 修改时间列：右对齐，统一格式 `YYYY-MM-DD HH:MM`；解析失败原样显示原始字符串，同样裁剪到列宽
  - 目录与文件用不同颜色/样式区分（有 `curses` 颜色支持时用配色，类似 `ls`；无颜色支持退化为仅用 `A_BOLD`）；选中行整行反色高亮
- 底部：常驻按键提示行，用于帮助提示 / 下载进度条 / 错误信息 / 覆盖确认交互（通过居中高亮的临时提示覆盖显示）；有颜色支持时用固定的青色粗体，不用 `A_DIM`（会降低对比度，暗色主题下可能看不清）；无颜色支持时退化为 `A_BOLD`；临时提示文字超宽同样中间省略（原则 3）
- 常驻按键提示行本身（"↑/↓/j/k 移动 ... q/Q 退出"）全长 114 列，同样中间省略（原则 3）

## 交互

| 按键 | 行为 |
|---|---|
| `↑`/`↓`、`j`/`k` | 移动选择 |
| `PageUp`/`PageDown` | 翻页 |
| `Enter`、鼠标左键点击某行 | 目录→进入；文件→下载 |
| `→` | 仅目录→进入；选中文件时不做任何操作（不触发下载） |
| `Backspace`/`←`/`u`/`Esc` | 返回上级（历史栈弹出） |
| `r`/`R`/`F5` | 刷新当前目录（重新请求并解析，不影响历史栈） |
| `q` | 退出 |

鼠标点击通过 `curses.mousemask(curses.ALL_MOUSE_EVENTS)` 启用，依赖终端本身支持鼠标上报（如 iTerm2、多数现代终端模拟器）；不支持时键盘操作不受影响；终端太小（见错误处理一节）导致列表根本没画出来时，点击会被直接忽略，不会误触发看不见的那一行；`curses.mouseinterval(0)` 见设计原则 1。

## 下载行为

- 本地文件名 = `unquote(href)` 的 basename，经原则 2 的控制字符净化；副作用是两个仅控制字符不同的服务器文件名会净化成同一个本地文件名，此时后一个下载会撞上前一个已存在的文件，走下面的覆盖确认流程（不会静默覆盖，原则 4）。basename 为空或是 `.`/`..`（畸形/构造的 href）时直接状态栏报错拒绝下载，不进入覆盖确认流程
- 目标路径已经是目录：直接状态栏报错拒绝下载，不进入覆盖确认流程（原则 4）
- 目标路径或其 `.part` 暂存文件已存在：弹出覆盖确认（原则 4）；交互细节（y/n 契约、Ctrl-C/resize/鼠标事件如何处理）见错误处理一节
- 触发下载后、真正发起网络请求前，先立即显示 `正在下载 file.txt ...（Ctrl-C 取消）`（原则 1）
- 下载中状态栏显示进度条 + 已下载/总大小 + 已用时（`M:SS` 格式，从下载开始计时）：`下载中 file.txt [=====-----] 45% 1.2M/2.7M 用时0:12`（无 `Content-Length` 时只显示已下载字节数 + 用时，不显示百分比/总大小）；进度回调按时间节流重绘，避免高频重绘导致闪烁
- 下载成功后状态栏提示 `已下载到 <本地路径>（用时 M:SS）`，返回浏览界面，光标停留在原条目
- 状态栏的临时提示（加载失败、"已在根目录"、下载取消/失败/成功等）以居中高亮的形式短暂显示，一段时间后自动消失，恢复为常驻的按键提示行；不影响下方已加载的目录列表

## CLI 参数

```
python3 nginx_tui.py <URL> [--output-dir DIR] [--insecure]
```
- `URL`：必填；缺少协议前缀（`http://`/`https://`）时自动补 `http://`
- `--output-dir`/`-o`：可选，默认当前工作目录
- `--insecure`/`-k`：可选，跳过 HTTPS 证书校验（用于自签名证书的服务器），默认不开启；一旦开启，加载/刷新/下载都不校验证书，有中间人攻击风险，仅应在信任目标网络时使用
- 不带任何参数运行（`python3 nginx_tui.py`）：直接打印完整帮助信息（等价于 `--help` 的输出：用法、参数说明、示例）并退出，不是 argparse 默认的简短报错
- `-h`/`--help`：同上，随时可查看帮助

## 错误处理

- 目录加载失败（网络错误、HTTP 非 200）：状态栏报错，停留在当前目录，不写入历史栈；页面解析不到任何链接则当作空目录处理，不算失败（见架构与组件一节）
- 加载/刷新目录列表、下载文件时，状态栏提示文字都带「（Ctrl-C 取消）」的提示；加载/刷新提示里显示的是目录/文件的短名字而不是完整 URL（进入子目录时取该条目的名字，刷新/启动时取 URL 最后一段路径），避免嵌套很深的路径把关键信息挤出屏幕；这个短名字始终带末尾 `/`——进入子目录时天然来自带斜杠的 `Entry.name`；刷新/启动时用 `_dir_label` 取 `_url_label` 的结果并补回被丢掉的斜杠（`_url_label` 提取"路径最后一段"本身会丢斜杠）——确保同一个目录不管是"正在加载"还是"正在刷新"，提示文字看起来一致
- 加载（进入子目录）/刷新目录列表等待期间按 Ctrl-C：取消（原则 5，保留旧状态）；应用启动时的首次加载被取消则打印取消信息并以退出码 130 退出（与其他 Ctrl-C 退出路径一致），区别于加载真的失败（网络错误等）时的退出码 1
- 空闲浏览（没有加载/刷新/下载/覆盖确认在等待中）时按 Ctrl-C：不退出程序，直接忽略；退出程序只能用 `q`/`Q`；这个保护覆盖主循环一整轮按键处理（画面重绘、读键、动作分发全部在同一个 try 里），不只是读键那一步，移动/翻页/返回上级这些没有自己 Ctrl-C 处理逻辑的动作也不会被 Ctrl-C 打断退出
- 下载失败/中断（Ctrl-C、断网、磁盘写入失败/权限不足）：见原则 4，报错后回到可操作的浏览界面
- 加载/刷新/下载这类会阻塞等待网络的操作结束后，会丢弃等待期间积压的按键/鼠标输入，不会在操作完成的瞬间把用户在等待时随手按的键/点的坐标"补放"出来当成新的操作
- 覆盖确认提示（y/n）等待期间：按 `y`/`Y` 才开始下载，按其他任意键（含 `Esc`）取消并回到浏览界面、不报错（`Esc` 在这里只表示取消，不会触发返回上级目录）；按 Ctrl-C 同样视为取消，不退出程序；期间收到终端 resize 会重新收敛滚动位置并重绘，收到鼠标事件会正常消费掉（避免残留在鼠标事件队列里影响后续点击）——这三者都不算作按键回答，继续等待真正的 y/n
- 启动参数错误（`--output-dir` 无法创建，如权限不足）：进入 TUI 前打印错误信息到 stderr，以非零退出码退出；`--output-dir` 支持 `~` 展开为用户主目录
- 终端 resize：捕获 `KEY_RESIZE`，重新计算布局并重绘（单帧绘制异常见原则 5）；resize 或刷新导致列表变短时，滚动位置会同时向下裁剪，不会停留在已经超出列表末尾的位置留下空白
- 终端窗口太小（高度 < 4 行，或宽度小于名称/大小/修改时间三列能不换行容纳的最小宽度）：不再尝试挤出一个会换行/被截断的列表，直接在屏幕正中央显示「窗口太小，按 q 退出」；这个分支不会绘制底部按键提示行，所以退出方式必须写进提示文字本身——否则用户在这个状态下无从得知还能做什么；`q` 在这个状态下依然正常生效（按键处理和绘制是分开的，不受当前画的是什么影响），只是不可见；提示文字本身刻意写得很短，保证在比正常状态窄得多的宽度下依然完整显示（否则空间已经不够的场景下，文字太长在临界宽度自己也会被截断到看不出意思）
